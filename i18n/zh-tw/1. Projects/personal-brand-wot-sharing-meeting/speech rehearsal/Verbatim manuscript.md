## 第二部分
> 6min, 5min, 5min, 7min, 6min

- 什麼是 Monorepo？
	- 它是一種原始碼管理的模式，形式上是將多個專案集中到單一儲存庫中統一管理；
	- 與之相對的是 Polyrepo 模式，這種模式下各個專案都有獨立的儲存庫；
	- **簡而言之**，Monorepo 就是將多個不同的專案以良好的組織關係，放到單一儲存庫中維護；
- 那麼 Monorepo 是如何降低多專案的維護成本的？它透過
	- 複用基礎建設，讓開發人員重新專注於應用程式本身
	- 程式碼共享，以低成本實現程式碼複用
	- 原子提交，使用自動化的多專案工作流程
- 首先是透過複用基礎建設，讓開發人員重新專注於應用程式本身：
	- 在傳統的 Polyrepo 模式下，每個專案都需要由開發人員個別建立和維護；而在 Monorepo 中，只需要一兩個開發人員負責建立 Monorepo 工程，所有專案都能在單一儲存庫中統一維護，透過複用一套基礎建設（例如 CI 設定、Lint 規則、建置腳本等），**從而**降低多專案的維護成本
	- 此外，複用基礎建設也使得統一改造和升級基礎建設變得更加方便，例如想在 CI 流程中為所有專案加入型別檢查以提升專案品質，在 Polyrepo 下需要逐一修改每個專案，**這樣的成本其實非常高**。而在 Monorepo 下，基礎建設的調整與維護，能夠輕鬆地套用到多個專案中。
- 其次是透過程式碼共享，讓開發人員能夠以低成本複用程式碼：
	- 在 Polyrepo 中，維護公用模組的成本相當高，首先是除錯非常繁瑣，公用模組的除錯需要手動執行 npm link 與當前除錯的專案關聯起來，若公用模組較多，npm link 的步驟將極為繁瑣低效。
	- 其次，公用模組的升級也很繁瑣，需要手動管理依賴關係，先升級底層模組，再發布，最後才能升級上層模組，**若中途出錯，還得重新走一遍這個流程**。
	- 而在 Monorepo 中，可以直接一鍵建立公用模組，上層模組一鍵引入公用模組進行開發和除錯，底層模組的變更能夠直接被上層感知，甚至不需要經過 link 和 npm 發布，大幅減少了重複工作，也大大降低了抽離新『複用程式碼』的成本，這讓開發者更願意進行此類抽離工作，進而提高了程式碼複用率。
- 再者，是透過原子提交，讓開發人員享受自動化的多專案工作流程
	- 若業務需求涉及多個專案，在 Polyrepo 模式中，例如需要修改圖中的三個專案，需先修改並提交底層模組、執行每個模組的 CI 流程，處理上層模組時還得更新底層依賴，再跑一次 CI 流程，整個流程非常繁瑣且不連貫。
	- 而在 Monorepo 中，我們可以直接一次性調整並提交多個專案，CI 和發布流程也都一次搞定，**從而將多專案的工作流程自動化。**
- 我們簡單做個總結：
	- 在 Polyrepo 模式下，每個專案都有各自的一套基礎建設，且程式碼複用困難、工作流程割裂；而在 Monorepo 中，我們可以讓多個專案共用一套基礎建設，方便地共享程式碼，並使用一致的工作流程。
	- **很多時候，一個團隊的多個專案並非相互割裂，而是彼此關聯的。Monorepo 可以很方便地將這些專案組織到單一大型儲存庫中統一維護，從而極大地降低了多專案的維護成本。**

## 第三部分
> 8min, 9min, 11min, 11min30

- 接下來分享 Bundler 和 Build System 的工程化實踐，因為無論是單倉專案還是多倉專案，隨著程式碼規模和子應用數量的增加，都會導致建置效能下降，**為了應對這種情況**，我們分別建設了 Bundler 和 Build System 工具。
	- 其中 Bundler 是為了解決單一巨石應用建置速度慢的問題；
	- 而 Build System 則是為了解決 Monorepo 下建置速度慢的問題。
- 在前端領域，Bundler 是一種工具，**用於將多個前端資源（例如 JS、CSS、圖片等檔案）打包到一個或多個檔案中，從而讓瀏覽器能夠直接執行**；
	- 常見的 Bundler 工具有 Webpack、Rollup、Vite、Parcel、Esbuild。
- 接下來介紹我們自研的 Rspack Bundler，它是一個基於 Rust 語言的高效能建置引擎，具備與 Webpack 生態系統的互通性。
	- 從這段介紹中，我們可以得知 Rspack 有兩個特性：一是高效能，二是與 Webpack 生態的相容性。
- 首先是第一個特性，即採用 Rust 語言實現。由於 JS 是單執行緒的，雖然在 JS 中也有一些實現並發的方式，但這些方法都有種帶著枷鎖跳舞的感覺，**而在 Rust 中我們能很好地支援並發特性**，因此我們將建置過程中的任務**利用並發特性來執行，這極大地提升了建置效能**；
	- 這兩張圖片是 Webpack 與 Rspack 在建置過程中的執行緒情況對比，可以明顯看出，Webpack 實際上只是單執行緒，而 Rspack 則能充分發揮多核 CPU 的優勢，極致地壓榨出效能。
	- 這是我們官網上的一張對比圖，在相同專案下，Rspack 只需 4.2 秒，而 Webpack 需要 34.8 秒
- Rspack 的第二個特性是與 webpack 部分相容，目前的實現可以理解為 webpack 的子集，這個子集涵蓋了大部分常用配置，能滿足日常業務開發的需求，**那麼，為什麼要與 webpack 生態相容？**
	- **首先，webpack 的外掛機制滿足了專案對客製化的需求**
	- **其次，複用 webpack 豐富的生態，相當於以最小的成本優化大型專案的開發體驗**
	- **最後，現有的 webpack 專案數量龐大，相容 webpack 能大幅降低遷移成本**
- 這是兩個典型業務上的收益：兩個專案原本 dev 啟動耗時約 5 分鐘，使用 Rspack 後只需約 20 秒；原本 HMR 需要約 20 秒，使用 Rspack 後只需約 1 秒，效能提升基本都在 10 倍左右
- 簡單介紹一下 Build System，它透過處理 Monorepo 下的專案依賴關係圖，並根據此關係圖調度建置任務；
	- 為什麼 Monorepo 下需要 Build System？因為 Monorepo 不只是將多個專案放到單一儲存庫而已，**它還需要借助 Build System 來管理整個程式碼倉庫中的多個專案，並根據專案之間的依賴關係進行建置；**
	- 常見的 Build System 工具有 Bazel、NX、Turborepo、Lage 等
- 接下來介紹我們自研的 Monorepo 工具中，如何實踐 Build System，我們透過
	- 支援「任務並行能力」，採用最大化的並行任務加速
	- 支援「多級快取能力」，對建置任務實現了多級快取
	- 支援「按需建置能力」，根據程式碼變更的影響範圍進行建置
- 如圖所示，根據子專案的依賴關係轉換成一個任務依賴圖，其建置順序必須符合這樣的要求：上層專案的建置，需要等待底層專案建置完成。
	- 我們看方式一，即透過串行排成 DEBCA，這符合建置要求；但其效能較低，例如 D 和 E 可以並行，所以第二種方式是將 DE 以及 BC 進行並行處理，如此便能將原本的 5 個步驟加速到 3 個步驟。
	- 此時我們發現任務 C 並不依賴任務 D 的完成，但在方式二中，任務 C 卻需要等待 D 和 E 都完成後才能開始執行。
	- 因此引出第三種方式：在任務 E 完成後，D 和 C 以並行方式執行；
- **當 Monorepo 子專案規模增大後，每次開發或上線都會涉及多個子專案，若每次都需要對這些子專案重新建置，將會極大地拖慢建置和部署速度**。
	- **我們提供了對建置產物進行快取的能力，能夠同時將產物快取到本地和遠端，當相關子專案沒有修改程式碼時，將會複用先前的建置產物以縮短建置時間**。
- **在按需建置能力的實踐上，我們支援依影響範圍執行 CI 流程，透過 git diff 對當前有改動的程式碼進行依賴分析，從而只建置受影響的專案，否則每次 CI 都會完整建置所有子專案**。
- 此處以一個簡單的 Monorepo 為例，**依賴關係如圖所示**
	1. 若一次性全量建置所有專案，耗時約為 17.72 秒
	2. 若僅修改了 component 模組，按需建置的話，只需建置 component、app1、app2，此時耗時 8.94 秒，節省約 50% 的時間
	3. 再來看看無快取的情況下，僅建置 App1、App2、App3，耗時在 10.77 秒到 16.94 秒之間
	4. 若有快取，例如 component、sdk、util 已建置過，再單獨建置 App1、App2、App3 時，建置耗時在 7.55 秒到 9.74 秒之間，約節省 45% 的時間
- 透過 Bundler 和 Build System 的建設，我們分別借助 Rust 語言的高效能和遠端建置快取能力，極大地加速了大型應用的建置速度，但這不僅僅是開發速度上的提升，同時也為業務帶來了兩大巨大收益
	1. 拉高了巨石應用的上限：使我們能夠開發更龐大、更強大的應用程式
	2. 加快迭代速度：使我們能夠更快速、更頻繁地進行 AB 測試和功能發布

- 所謂微前端，就是前端領域的應用分治解決方案，XX 公司的微前端實踐也經歷了 iframe、SPA、框架等階段；在實踐中遇到不少問題，為了進一步降低多人協作的成本，目前正進入新方案的探索階段。
- 接下來看看微前端的工程化實踐，新方案是如何降低多人開發的協作成本的
	- 首先，透過減輕基座負擔，將基座應用與業務邏輯解耦
	- 其次，採用細粒度的組合，在更細粒度的模組上獨立開發、部署
	- 最後，透過約定模組協議標準，我們搭建了模組中心，甚至可以結合低程式碼平台，從而帶來更高的模組複用率；並支援模組級別的灰度和 AB 能力；
- 它是如何減輕基座負擔的呢？傳統的微前端透過建立一個公共基座來承載公共邏輯，這種複用方式除了工具庫之外，往往導致不少業務邏輯耦合到基座中，造成基座更頻繁的改動和發布，也更容易出現更大的故障影響範圍和更容易失效的快取；
	- 這又讓子應用從獨立開發、部署重新回到某種程度上的相互依賴。因此，新方案透過兩種方式消除了這類基座的存在：一是消費機制，二是共享機制，前者一般用於複用業務邏輯，後者用於複用與業務無關的工具庫
- 那麼，新方案的機制是如何發揮作用的？
	- 傳統的微前端架構中，多個子應用之間是相對隔離的，往往透過一套沙箱機制來保證這種隔離性；
	- 但隨著前端子應用的規模增大、人員增多，如此粗粒度的隔離反而會制約每個子應用內外的人員協作；
	- 因此，我們新的應對方式是透過提供更細粒度的**模組消費與共享方案**，讓開發人員能夠以更小的單位進行獨立開發、測試、部署
- **模組協議標準約定了模組的元資訊，透過該協議在各平台之間流轉，以達到特定功能**
	1. 例如建置平台會依據配置文件建置出這份協議文件
	2. **例如部署平台會將元資訊轉換成包含具體 cdn 地址的資料，直接回填到 html 中下發，這步操作也是實現模組灰度與 AB 測試的前提**
	3. 再例如應用運行時也會依據這份協議動態載入模組
	- 有了細粒度組合與模組協議標準的**特性**，我們便能十分方便地建立一個線上模組中心，無論是業務相關的元件，還是業務無關的工具庫，都能**以極低的成本**在團隊內部，甚至跨團隊複用；
	- 此外，**基於這套機制**，我們也嘗試與低代碼平台配合，透過低代碼平台搭建出符合模組協議標準的元件並註冊為線上模組，能夠大幅提升業務開發效率。
- 這是一個典型的接入業務案例，接入後我們發現從建置耗時、部署耗時，乃至迭代速度和需求吞吐量都有了顯著提升。
